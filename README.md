# Менеджер паролей GophKeeper

Индивидуальный дипломный проект курса «Go-разработчик»

# О системе

GophKeeper представляет собой клиент-серверную систему, позволяющую пользователю надёжно и безопасно хранить логины, пароли, бинарные данные и прочую приватную информацию.

# Особенности менеджера
Создавать и изменять записи можно только при наличии связи с сервером. Чтение доступно и в офлайн режиме.
Для использования сервиса с другого устройства необходимо перенести клиентский ключ на данное устройство. 

# Взаимодействие клиента и сервера
## Создание записи
0. На клиенте создается запись в клиентском хранилище. Поля id и дата создания пустые.
0. Фоновое задание отправляет запись на сервер.
0. На сервере в серверном хранилище создается запись.
0. Сервер возвращает id и дату создания записи.
0. Клиент записывает в своем хранилище id, дату создания записи и номер версии = 1.

Если с разных клиентов одного пользователя будут созданы одинаковые записи(например, креды для одного и того же аккаунта), то на сервере появится эти 2 записи.
В будущем можно реализовать логику, которая будет "схлопывать" такие записи.

## Редактирование записи
0. На клиенте происходит запись отредактированных секретов в хранилище с пометкой о том, что необходимо обновить данные на сервере.
0. Фоновое задание отправляет данные на сервер для обновления.
0. Сервер принимает запись, по id находит запись в серверном хранилище, блокирует запись.
0. Сервер сравнивает номер версии с клиента с номером версии на сервере по данной записи.
0. Если версии равны, то обновляет запись и возвращает клиенту обновленный номер версии(+1) и дату обновления.  
0. Если версии не равны, то возвращает ошибку обновления данных - данные были изменены другим клиентом пользователя.
0. На клиенте у записи в клиентском хранилище снимается пометка о том, что необходимо обновить данные на сервере. Обновляются номер версии (брать с сервера или просто +1 делать?) и дата обновления.

После создания/обновления записи происходит обновление таблицы, в которой хранится номер версии данных пользователя и дата обновления.

## Получение изменений, сделанных на других клиентах пользователя.
0. Фоновое задание считывает версию данных из локального хранилища.
0. Дергает серверную ручку, в которую передает номер версии.
0. На сервере происходит сравнение поступившей версии с той, что в хранилище на сервере. 
0. Если поступившая версия меньше той, что на сервере, то выбирается дата, соответствующая данной версии(она хранится в отдельной таблице с версиями без привзяки к секретам).
0. Отбираются все секреты, дата обновления которых больше или равно данной дате. Отправляются на клиент вместе с версией и датой текущий данных юзера.
0. Если поступившая версия равна той, что на сервере, то возвращается пустая выборка.
0. Если поступившая версия больше той, что на сервере, то выдать ошибку.
0. На клиенте обновляются возвращенные с сервера записи и значение версии и даты текущих данных юзера.

Минус подхода - после создания/обновления записи на сервере будет происходит синхронизация данных и будут получены информация по этим же секретам (номер версии, дата создания и обновления).
Упрощенная версия - создание и обновление записи производить только онлайн, тогда такой проблемы не будет.
